# Model Construct — grok-console — §7 Policy Enforcement (alternate realization)

This is a **second data point** for the domain already worked in [`standard/part-2-domains/07-policy-enforcement.md`](../../standard/part-2-domains/07-policy-enforcement.md) §7.6, which uses thinx as its informative example. Read that clause first — this entry maps the same controls onto a structurally different mechanism, described at the pattern level rather than by implementation.

## Mapping

| AHES-PE requirement | thinx realization (existing §7.6) | grok-console realization |
|---|---|---|
| PE-A1 policy document | A policy document enumerating hard-stops, flagged actions, and trust scope | A comparable policy document, with an explicit **conflict-priority rule** for competing obligations not present in the thinx realization: obligations are ranked, and when two conflict the one with greater irreversibility risk wins, with an explicit default of pausing for human input rather than resolving silently |
| PE-A2 enforcement-point inventory | A dispatcher that intercepts tool calls before execution, implemented as harness-authored code | A **declarative pattern catalog** — deny rules expressed as data, not as dispatcher logic |
| PE-02 structural hard-stop | A hook intercepts before the model's chosen tool executes | The catalog's deny rules are loaded into the model CLI's own **native permission engine** at session start — enforcement lives inside the substrate's own permission system, not a wrapping interceptor. A structurally different mechanism achieving the same property: the model's own reasoning cannot execute a denied pattern regardless of what it argues for |
| PE-04 override, attributable | A per-action override marker with an append-only audit log | Deliberately **absent by design** for the hard-deny tier — override for that tier is human-mediated outside the harness entirely, not a flag the harness itself exposes. A separate, lower-severity tier (see below) does carry an attributable marker-and-log pattern, scoped narrower than thinx's |
| — (three-tier extension not named in the base AHES-PE set) | Binary allow / hard-stop, with soft-stop as a judgment-layer convention | An **explicit three-way outcome contract**: allow, confirm, or deny. Confirm is a first-class enforcement tier, not merely "flag and proceed" prose — commands in this class require a separate structural checkpoint distinct from the deny mechanism, because the native permission engine's deny is evaluated at session boundaries, not continuously |
| Evidence binding (→ §10) | An audit log for overrides | Enforcement decisions can optionally be written into the same evidence architecture used elsewhere in the harness, tagged with the authority class the decision fell under |

## What's different from other exemplars

The load-bearing difference: thinx's enforcement point is **harness-authored code** that intercepts tool calls before the model's chosen CLI ever sees them. grok-console's is a **catalog consumed by the model's own CLI's native permission system** — the harness doesn't intercept, it configures the substrate's built-in gate. Both discharge AHES-PE-02 (structural, not model-reasoning-dependent), but they demonstrate the requirement is achievable through two architecturally opposite paths: wrap-and-intercept vs. configure-the-native-gate. A standard whose PE-02 wording only fits one of these patterns would be over-fit to thinx; this exemplar is the check that it isn't.

grok-console also names a control neither the base clause nor thinx's realization states explicitly: a **documented conflict-priority rule** for competing obligations. This motivated `AHES-PE-10`, now a real requirement in `standard/part-2-domains/07-policy-enforcement.md` §7.3 — though the two rules are not identical: grok-console's own tiebreak criterion is *risk-magnitude-based* (the obligation with greater irreversibility risk wins), while PE-10 adopted a *tier-based* ordering (prohibitions outrank obligations outrank permissions) as AHES's own normative choice, judged more decidable for verification purposes than a magnitude comparison. Both share the same same-tier default: pause for human input rather than resolve silently. Treat grok-console as the exemplar that established *a conflict-priority rule belongs in this domain*, not as a description of PE-10's exact mechanism.

## Gaps

- The confirm tier's structural enforcement is disclosed by the source repo itself as incomplete for one sub-case (a mid-turn interruption path is not yet closed) — cited here at the pattern level rather than by exact tracking-item reference. The source is honest that PE-02-equivalent structural enforcement is not yet complete for every case in this tier.
- For at least one higher-risk approval class, the source repo documents that attribution fields (reason, blast radius, rollback plan, approver) are policy convention rather than a structurally enforced marker gate — i.e., PE-04's attributability requirement is, for that class, currently a documented convention rather than mechanically enforced the way the deny tier is.
