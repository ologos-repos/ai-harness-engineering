# Model Construct — thinx-codex — Part 5 Conformance & Assessment

This entry grounds **Part 5** — the conformance-levels and assessment-procedure part — rather than a Part 2 domain. Read [`standard/part-5-conformance.md`](../../standard/part-5-conformance.md) first: this maps the §5.5 assessment procedure and §5.2 level structure onto a real, running mechanism, described at the pattern level rather than by implementation.

thinx-codex maintains an authored **trace spine** — strategic intent, requirements, architecture, and verification, held as authored artifacts with a fixed ID vocabulary — and runs a family of gate scripts over it in continuous integration. The gates read the authored baselines and generate nothing; they only check integrity and fail the build on violation. That is a working instance of "conformance verified, not asserted," and it maps onto Part 5 as follows.

## Mapping

| AHES Part 5 concern | Pattern observed in thinx-codex | How it is satisfied / what it precedents |
|---|---|---|
| §5.2 level structure — **structural (`shall`) vs advisory (`should`)** | The conformance contract splits its rules into two tiers: a **fail-closed** tier whose violation blocks the change, and a **fail-open** tier whose violation is *reported* but does not block while a revision is still in progress | A direct architectural precedent for AHES's L1/L2/L3 ladder and the Part 0.2 `shall`/`should` split. The fail-closed tier is what an L2 (Enforced) claim means made executable — a structural control that operates, not a documented intent; the fail-open tier is the `should`-grade coverage recommendation surfaced without gating. AHES's level model and this split are the same idea reached independently |
| §5.5 step 3 — **requirement selection over a traceable set** | Requirements, objectives, architecture nodes and verification items each carry an ID in a `<KIND>-<AREA>-<NNNN>` convention, drawn from a fixed, small vocabulary of kinds (objective / key-result, requirement, architecture node, interface, verification item) | An independent instance of the exact discipline AHES assumes for its own `AHES-<domain>-<nn>` scheme (Part 0.2). Because every artifact is addressable by a stable ID of known kind, "select the binding requirement set" becomes a mechanical graph query rather than a manual reading exercise — evidence that §5.5 step 3 is automatable given the same ID discipline the standard already mandates |
| §5.5 step 2 — **applicability matrix / no silent omission** | A structural rule family enforces **no orphan** (every architecture node satisfies at least one requirement), **no unallocated requirement** (every requirement traces up to an objective or key result), **referential integrity** (every trace edge resolves to an authored ID), and **acyclic dependency** (the architecture dependency graph is a DAG) | This rule family is a candidate structural realization of §5.5's "silent omission is a gap, not an inapplicability" (AHES-CF-05). An unallocated or orphaned node is exactly the machine-detectable form of a domain claimed but not grounded, or an artifact present but tied to nothing. AHES's applicability matrix and requirement-selection rule could be checked the same way: every applicable domain must resolve to a populated, referentially-intact set of artifacts, or the assessment fails closed rather than passing by omission |
| §5.5 step 4-5 — **evidence review → disposition, run mechanically** | The gate family runs on every pull request and every push to the main line through a CI workflow; the driver runs each structural gate in sequence and **short-circuits on the first failure**, so a single unmet structural rule fails the whole run and blocks merge. A diff-scoped evaluator step runs additionally on pull requests | A precedent for what an AHES **L2/L3 assessment tool** could look like: the "disposition" (§5.6 Pass / Fail) is the CI exit status; "evidence sampling" (§5.5 step 4) is, for the structural tier, full-census rather than sampled — which is exactly what AHES-CF-06 requires for hard-stop / non-overridable classes (census, not sampled). The advisory tier is where AHES's risk-weighted sampling would sit |
| §5.5 — **the assessment tool's own coverage** | A self-referential meta-gate enforces that *every* gate script is actually wired into the CI workflow and named in the harness's own tradecraft surface — no gate that exists but never runs, no enforcement documented but not executed | The sharpest precedent of the set. An assessment procedure is only as trustworthy as its own completeness; here the tooling gates *itself* for coverage. AHES §5.5 has no analogue yet — a check that an assessment actually exercised every applicable domain rather than quietly skipping one is the natural forcing-function this pattern suggests |
| §5.2 — **one control per concern area** | Enforcement is decomposed into one gate per governance layer — a trace-conformance gate over the spine, plus sibling gates for metadata discipline, the memory store, the execution/tool surface, the tradecraft surface, the program registry, and the planner/generator/evaluator wiring | A precedent for structuring an AHES conformance tool by clause/domain: a gate per Part 2 domain, composed by a driver, is a clean realization of AHES-CF-01/02/03's per-domain-section selection rule (§5.4). The decomposition also localizes findings to the domain they trace to, which is what §5.6 disposition-by-requirement-ID needs |

## What's different from other exemplars

Every other entry in `model-construct/` demonstrates a **runtime enforcement** mechanism — grok-console's policy catalog gates tool calls as the model runs; mxh-p's envelope carries governance into each spawn; thinx-Claude's hook intercepts before execution. thinx-codex is the only exemplar that demonstrates **build-time, automated conformance checking**: the control does not mediate a live action at all. It reads the *authored governance artifacts of the harness itself* and blocks a change to them when their structure is unsound.

That is a categorically different data point, and it is the one Part 5 specifically needs. Part 5 is about assessing whether a harness *is what it claims to be* — whether its artifacts exist, trace, and cohere. thinx-codex shows that this assessment can be mechanized as a fail-closed CI gate over a traceable artifact graph, rather than performed only as a human documented-evidence review. It is the missing worked example of §5.5 realized as tooling instead of prose.

## Gaps

- **Reliability of the gates was not verified.** This review confirmed the gate family exists, is CI-wired, and enforces the described rule families. It did **not** trace whether the checks are bug-free, whether their rules are complete against the trace schema, or whether a malformed baseline could slip past a given rule. "The gate is fail-closed" is a claim about intent and observed structure, not a proof of soundness.
- **Not every gate was inspected individually.** The per-concern gates beyond the trace-conformance and CI-driver layers were read only far enough to characterize each one's governance concern. A per-gate mapping (which structural invariant each enforces, and how completely) is a natural follow-up, not attempted here to avoid an ungrounded claim.
- **The fail-open tier's downstream behavior is described from its contract, not observed in a run.** How the advisory (coverage / provenance / realization-backlog) findings are surfaced in practice, and whether they are ever escalated to blocking, was not exercised against a live CI run.
- **This is an architectural-pattern read, not a conformance assessment of thinx-codex against AHES.** It maps thinx-codex's *own* conformance machinery onto Part 5 as a precedent. It does not assert that thinx-codex conforms to AHES at any level — that would require running AHES's own §5.5 procedure against it, which is out of scope here.
