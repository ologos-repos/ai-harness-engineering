# Model Construct — mxh-p (claim-verification rigor, with portability as a consequence)

**Subject:** `ologos-repos/mxh-p` (private) — a reference implementation of a governance-synthesis pattern from the AIDE canon, shipped as an installable package with a CLI entrypoint.

**Correction (2026-07-16).** An earlier version of this entry characterized the package primarily as a §16 Substrate Portability mechanism, with its epistemic-labeling contract noted as a secondary feature. That got the design center backwards. The package's own claim-ledger schema is explicit: a unit of work cannot be marked delivered unless it went through an actual planned → generated → skeptically-evaluated execution chain — a mode of "artifact-only" (files exist but the verification chain doesn't) is structurally prohibited from claiming success. Combined with the orientation payload's required epistemic-labeling contract (claims must be tagged as computed, inferred, or uncertain; assertions require grounding), the package's primary purpose is **driving deterministic, verifiable outcomes and structurally refusing unverified claims of completion** — an anti-hallucination discipline enforced by harness rigor, not by asking the model to be more careful. That the resulting artifact happens to be a portable, versioned package is a real and useful property (§16 below), but it is a consequence of good packaging discipline, not the thing the package was built to do.

**Why this exists, separately from grok-console's entry.** [`model-construct/grok-console/16-substrate-portability.md`](../grok-console/16-substrate-portability.md) documents one *consumer* of this kernel's portable envelope. This entry documents the kernel *itself* — both its primary claim-verification mechanism and the portability property that follows from it.

**Scope discipline, briefly.** The package is deliberately narrow: it ships the orientation/governance-envelope layer and the claim-verification ledger, and explicitly does not absorb an adjacent multi-worker coordination runtime that lives in a separate repository. A kernel that resists scope creep into adjacent concerns is more likely to stay both portable and its claims trustworthy — worth noting as a pattern in its own right.

## Domain realized (primary): §11 Evaluation — claim-verification as a structural gate

| AHES clause concern | Pattern observed | How it's satisfied |
|---|---|---|
| Adversarial / skeptical acceptance, not self-report | A claimable delivery ledger requires a specific execution mode (an actual dispatched multi-role chain — plan, generate, evaluate) to succeed; a mode representing artifacts existing without that chain having run is structurally prohibited from claiming success | This is §11's adversarial-evaluation concern realized as a **fail-closed schema constraint**, not a review policy: the harness cannot be talked into accepting an unverified completion claim, because the claim ledger's own schema won't validate one |
| Epistemic labeling as a required contract | The governing payload injected at every spawn requires claims to be tagged by epistemic status (computed vs. inferred vs. uncertain) and requires grounding before assertion | This is evaluation discipline pushed *upstream*, into the generation step itself, rather than caught only at review — the harness structurally refuses to treat an untagged or ungrounded claim as equivalent to a verified one |
| Distinct roles for planning, generation, and evaluation | The claim ledger requires artifacts from three separate roles (a planning artifact, a generation artifact, an evaluation artifact) before a unit of work can be marked delivered | Structural separation of "did the work" from "checked the work" — the same role a review role plays in this standard's own §4 Agent engineering informative examples, but here made a *schema-level* precondition for claim success rather than a workflow convention |
| Deterministic, reproducible claim status | Delivery status is derived from the ledger's recorded execution mode and per-role dispatch records, not from a free-text summary of what happened | The claim is checkable mechanically — an assessor (or an automated gate) can validate a delivery claim from the ledger alone, the same evidentiary posture this standard's own §10 Evidence engineering and Part 5 assessment procedure require |

## Domain realized (secondary): §16 Substrate portability

| AHES clause concern | Pattern observed | How it's satisfied |
|---|---|---|
| A portable, schema-defined governing payload | The kernel defines a strict schema for the governing envelope injected at every spawn, with required top-level sections corresponding to each MxM construct surface (mission, reasoning posture, constraints, tradecraft, memory, execution substrate) | Functions as **AHES's own domain vocabulary expressed as an enforced schema** — independent cross-check that the construct vocabulary this standard assumes is already load-bearing in shipped tooling |
| Explicit authority scoping carried across spawns | A spawned worker's authority is represented as an explicit, expirable, attributed grant object inside the portable payload itself | The mechanism-level answer to §9's principal-and-authority-model concern, expressed as data rather than convention |
| Epistemic discipline carried across spawns | The same labeling contract that drives §11's claim-verification (above) travels with the payload across any substitution boundary | Portability of *reasoning discipline*, not just permissions — most governance-porting efforts port rules and skip this |
| Distribution + version discipline | Ordinary package-engineering practice (versioning, install, a documented conformance mapping back to its own upstream pattern) applied to a governance artifact | What makes the portability claim testable rather than asserted |

## What's different from other exemplars

Every other exemplar in this standard (thinx, thinx-codex, grok-console's own construct) is a **deployment** — a harness doing its own governance. This kernel is **infrastructure for claim-verification and governance-porting** — a package whose primary purpose is refusing to let unverified work count as done, with the portable-envelope property as a secondary, genuinely useful consequence of how it's packaged. It's the clearest available evidence in this standard that both AHES §11 (evaluation as structural gate, not review policy) and §16 (portability as buildable artifact, not aspiration) describe artifact classes that actually get built this way.

## Gaps

- The claim ledger's fail-closed behavior was verified from its schema (the required execution mode and per-role artifact fields), not from a live test of the gate rejecting an artifact-only submission — a fuller write-up would want to see that rejection actually exercised, not just structurally implied by the schema.
- The package's own scope is explicit and narrow: a fuller multi-worker coordination runtime, cross-team federation, and a larger project-lifecycle layer are all named out of scope for the current version. It demonstrates claim-verification and portability for the *orientation and single-package-delivery* slice, not the full sixteen-domain surface this standard specifies.
- This entry does not address §16's "bounded, specified behavioral change" measurement requirement (AHES-SP-06) — see [`model-construct/grok-console/16-substrate-portability.md`](../grok-console/16-substrate-portability.md) for that gap, which remains open on the consuming side.
