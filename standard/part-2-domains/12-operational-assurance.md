# AHES Part 2 §12 — Operational Assurance

**Status: Draft v0.1 — worked clause (normative shape complete; requirement set subject to review)**

## 12.1 Purpose

Operational assurance governs the running system: monitoring, drift detection, anomaly detection, rollback, and incident response for harness-specific failure modes — model regression, tool breakage, and policy gaps discovered in production. When this domain is unengineered, the failure class is **silent degradation**: behavior shifts and nobody's dashboard says so, so a regression accumulates cost until a human happens to notice. Operational assurance is the *post-deployment* counterpart to evaluation (→ §11): where §11 qualifies a change before it ships — *does this meet the bar* — §12 answers *is the running system still meeting the bar, and what happens when it stops*. The seam between them is the baseline: the passing result set §11 produces at qualification (→ §11 EV-06) is the reference §12 monitors against, so drift is defined as measured divergence from a qualified baseline, not as an operator's intuition that something feels off. This clause is the harness-side realization of the **model drift and silent regression** risk in the risk register (→ Part 4).

## 12.2 Required artifacts

- **AHES-OA-A1** — A **monitoring specification** naming the signals the harness monitors in production — output-distribution measures against the qualified baseline (→ §11 EV-A2), error and tool-failure rates, policy-denial and override rates (→ §7), latency and cost, and evidence-emit completeness (→ OA-10) — each with a threshold and the accountable owner alerted when it is crossed. Owner: harness operator. Change control: versioned with the configuration baseline (→ §15); thresholds are baseline configuration, not ad-hoc dashboard settings.
- **AHES-OA-A2** — A **drift/regression detection definition**: the method by which running behavior is compared to the §11 baseline, the divergence measures used, and the thresholds at which drift is declared. Owner: harness operator. Change control: versioned; a threshold change is a baseline change.
- **AHES-OA-A3** — A **rollback runbook** naming the rollback targets (prior qualified baselines under §15), the trigger conditions, the accountable authority who may invoke a rollback, and the expected recovery behavior. Owner: harness operator. Change control: versioned; exercised, not merely written (→ OA-06).
- **AHES-OA-A4** — An **incident response plan** for harness-specific failure modes — model regression, tool breakage, policy gap discovered in production — defining severity classes, escalation paths, accountable owners, and containment options (fail-closed, degrade, halt). Owner: harness operator (security-reviewed for policy-gap class). Change control: versioned.

## 12.3 Technical controls

- **AHES-OA-01** — The running harness **shall** be monitored against the qualified baseline (→ §11 EV-A2), not against a separately hand-set operational target. The reference for "normal" **shall** be the same result set §11 produced at qualification; monitoring against a divergent baseline reintroduces the conflation §11's shared-baseline requirement (EV-06) exists to prevent.
- **AHES-OA-02** — The harness **shall** detect drift — measured divergence of production behavior from the qualified baseline, including model regression after a substitution or a provider-side change (→ §1) — and **shall** raise it to the accountable owner rather than silently absorb it. Drift below threshold **may** be recorded without alert; drift crossing the OA-A2 threshold **shall** alert. This control is the harness-side mitigation the risk register maps to §12 (→ Part 4, *model drift and silent regression*).
- **AHES-OA-03** — Runtime anomaly signals — error-rate spikes, tool-failure bursts, policy-denial or override surges (→ §7), latency or cost excursions — **shall** be monitored against defined thresholds (OA-A1), and a crossing **shall** raise an alert attributable to an accountable owner. Anomalies **shall not** be detectable only in retrospect from raw logs; the threshold and alert are the control.
- **AHES-OA-04** — Tool breakage — postcondition failures, contract violations, or external-dependency failures (→ §3) — **shall** be detected and surfaced, not swallowed as a generic error. A tool that has silently begun to fail or to return degraded results **shall** raise an anomaly, because a broken tool the harness keeps invoking is a capability the mission is relying on and no longer has.
- **AHES-OA-05** — A **policy gap discovered in production** — an action that should have been gated but was not (→ §7) — **shall** be handled as an incident under OA-A4, and **shall** feed back into the policy baseline (→ §7) and the adversarial evaluation corpus (→ §11 EV-A3), so the same gap cannot recur unqualified. Discovering a gap and closing only the instance, without landing the regression case that gates it, is an incomplete response.
- **AHES-OA-06** — The harness **shall** support rollback to a prior qualified baseline (→ §15) without code change, and the rollback path **shall** be exercised, not merely documented. For a model regression, this is the §1 rollback path (MM-08); for an instruction or policy regression, the corresponding baseline revert. Rollback trigger conditions **shall** be defined (OA-A3) and every rollback **shall** be recorded as an attributable event (→ §12.5).
- **AHES-OA-07** — Harness-specific failure modes **shall** have a defined incident response with an accountable owner, a severity classification, and containment options — fail-closed, degrade to a reduced-capability posture, or halt side effects. Incident response **shall not** depend on the model's cooperation for containment: a failure whose cause may be the model itself is contained structurally (→ §7), consistent with the domain's threat model.
- **AHES-OA-08** — On detected drift or anomaly exceeding its threshold, the harness **shall** default to a safe posture — reduce capability, escalate to a human authority (→ §8), or halt side-effecting actions (→ §7) — rather than continue operating an unqualified configuration. Continuing full-capability operation while a monitored measure is out of bound is the silent-degradation failure the domain exists to prevent, now with the signal available and ignored.
- **AHES-OA-09** — Incidents and near-misses **shall** feed the §11 evaluation suite: each closed incident **shall** yield a regression case added to the evaluation regime (→ §11 EV-A1/EV-A3), so that a production failure becomes a pre-deployment gate and cannot re-ship. This closes the loop opened by §11 — evaluation qualifies what ships; operations qualifies what evaluation must henceforth cover.
- **AHES-OA-10** — The harness **shall** monitor for evidence-emit failure: accountable actions (→ §10 EE-01) that did not produce a validated evidence record. This closes the gap §10.5 explicitly defers here — the evidence schema and write-time validator (→ §10 EE-02) guarantee that *committed* records are well-formed, but guarantee nothing about whether every accountable action actually attempted a write. A detection method **shall** exist independent of the evidence store's own contents (e.g., comparing an independent count of accountable-action invocations against the corresponding evidence-record count, or instrumenting the action path itself), since a store that silently failed to receive a record cannot be relied on to report its own absence. A detected gap **shall** be handled as an incident under OA-A4.

## 12.4 Verification criteria

| Control | Method |
|---|---|
| OA-01 | Inspection: the monitored baseline reference is the §11 qualification result set (OA-A1 ↔ §11 EV-A2), not a separate target. Analysis: confirm no divergent operational baseline exists. |
| OA-02 | Test (fault injection): introduce a controlled behavioral regression or model substitution; confirm drift is measured against the baseline, crosses threshold, and alerts the accountable owner. |
| OA-03 | Test: inject anomaly conditions (error bursts, denial surges, latency excursions); confirm each crosses its threshold and raises an attributable alert. Inspection of threshold definitions against OA-A1. |
| OA-04 | Test: induce a tool postcondition/dependency failure; confirm it surfaces as an anomaly rather than being swallowed. Inspection of tool-failure handling against §3 contracts. |
| OA-05 | Demonstration: walk a discovered policy gap through the incident path; verify both the policy baseline (§7) and the adversarial corpus (§11 EV-A3) are updated, not just the instance. **Adversarial test**: re-run the gap case after closure and confirm it is now gated. |
| OA-06 | Demonstration: exercise a live rollback to a prior qualified baseline with no code change; confirm behavior returns to the known-good baseline and the event is recorded. Inspection that the runbook (OA-A3) is exercised on a defined cadence. |
| OA-07 | Inspection of the incident response plan (OA-A4) for accountable owners, severity classes, and containment options. Demonstration: run an incident of each failure class through containment. |
| OA-08 | **Adversarial test**: drive a monitored measure past its threshold and confirm the harness enters a safe posture rather than continuing full-capability operation; verify the safe posture cannot be bypassed by model reasoning (→ §7). |
| OA-09 | Analysis: trace a sample of closed incidents to the regression cases they produced in the §11 suite; confirm the loop is closed (every incident yields a gating case or a recorded rationale for none). |
| OA-10 | **Adversarial test**: induce an accountable action whose evidence write is suppressed or fails silently (independent of the evidence store's own reporting); confirm the gap is detected and raised as an incident, not left undetected. Inspection of the detection method's independence from the evidence store it's monitoring. |

## 12.5 Evidence requirements

The operating harness **shall** capture: the monitored signals and their values over time against the qualified baseline, every threshold crossing (measure, value, threshold, timestamp), every drift and anomaly declaration with the owner alerted, every rollback (target baseline, trigger, invoking authority, timestamp), and every incident with its classification, containment action, and the evaluation regression case it produced (→ OA-09). This is sufficient to answer, post-hoc: *when did behavior diverge from the qualified baseline, was it detected, what was done, under whose authority, and did the fix become a future gate*. Evidence flows into the §10 evidence architecture; the monitored baseline and the configuration baseline active at each incident are retained under the configuration-baseline retention rule (→ §15), so that an incident review can reconstruct which baseline was running when behavior drifted.

## 12.6 Informative examples

**Monitoring-and-flag surface (MxM reference architecture).** A multi-mode meta-harness realizes drift detection (OA-02) as a durable signal feed rather than an ephemeral impression: a model-fit mismatch — the running model diverging from the shape of the work in front of it — is logged to an append-only record that an operator console panel surfaces, so the signal survives past the session that noticed it. A session-start readiness report is realized as the recurring monitoring surface (OA-01/OA-03): daemon state, version-control state, and recent-activity ages presented against expected-normal each session. Rollback (OA-06) is the baseline-revert path shared with model mediation (→ §1): a regressing model or instruction change reverts to a prior pinned baseline without code change. The architectural point: the baseline these surfaces monitor against is the one evaluation qualified (→ §11), and a production incident feeds back as a new evaluation case (OA-09), so the two domains form one closed loop rather than two disconnected phases.

**General pattern.** OA-08's safe-posture-on-threshold requirement follows from the domain's failure class: silent degradation is not the absence of a signal but the presence of one that changes no behavior. A monitored measure out of bound that leaves full-capability operation untouched has reproduced the failure the monitoring was installed to prevent, now with the evidence in hand and unacted-upon.
